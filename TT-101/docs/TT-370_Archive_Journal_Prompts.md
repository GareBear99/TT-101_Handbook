## Purpose

The TT handbook includes **journal prompts only where they produce meaningful, verifiable records**:
- actions you can perform **now**
- artifacts you can generate **now**
- checks you can repeat **later**
- outputs that can be validated (hashes, tokens, files)

This avoids turning speculative sections into “proof claims.”

## What counts as meaningful to record

Record:
- build / validation runs (pass/fail, tool, version)
- manifest hash or release id
- beacon drafts + publication locations (hypothesis-only)
- slate pack/unpack parameters (chunk size, parity plan)
- training completions (decode puzzles solved)

Do not record:
- claims of future contact or proof
- coercive “if you don’t respond…” language
- personal sensitive information

## Journal line format

Each journal entry should include:
- Segment ID (e.g., TT-315)
- Title (short)
- Body (factual)
- Evidence (optional): manifest hash, token, file path

## Continuity token

Entries are chained into a continuity token (see TT-368):
- token changes with each appended entry
- token is reproducible given the same entries in order
- token can be etched/printed as a **compact continuity proof**

## Segments with built-in prompts (current allowlist)

- TT-201 (continuity protocol actions)
- TT-250 / TT-255 / TT-258 / TT-259 (beacon + observer + divergence + experiment)
- TT-315 (binary QR slate artifact steps)

These prompts are **optional** and should be used only when you actually performed the step.
